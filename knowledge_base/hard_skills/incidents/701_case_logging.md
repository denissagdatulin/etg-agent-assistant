## Индикатор: Оформление и логирование кейса в системе
- Тип: Hard

### Когда индикатор применим

Индикатор применим всегда.

Оценивай только финальное состояние уже закрытого тикета:

- блок `Information`;
- достаточность комментариев;
- наличие значимых вложений, переписки, звонков и согласований, если без них ход решения нельзя понять.

Не оценивай `701` по промежуточным изменениям как по отдельным нарушениям, если финальное закрытое состояние тикета уже оформлено корректно и читается однозначно.

### Допустимые оценки

- `0` — три и более независимые ошибки логирования или критически неполное оформление
- `1` — две независимые ошибки логирования
- `2` — логирование и оформление в целом корректны

### Что именно проверять

- `TIMEFRAME`
- `POOL`, `STATUS`, `GUEST STATUS`
- `CATEGORY`
- `TYPE`, `SUBTYPE`, `SOURCE`, `CAUSE`
- `BOOKING OUTCOME`
- `REPORTED CHANNEL`, `REPORTED BY`
- `INTERNAL BOOKING STATUS`, если оно применимо
- достаточность `AGENT COMMENT`
- наличие значимых звонков, переписки, вложений и согласований

Дополнительно учитывай:

- если тикет создал оцениваемый сотрудник, оценивай корректность `ORIGINAL MESSAGE`;
- если поле `Client Involved` было проставлено поздно и из-за этого поздно изменился связанный статус клиента, считай это одним логическим дефектом, а не набором отдельных ошибок;
- в кейсах с согласованной компенсацией должно быть видно, кто и где согласовал потери, если такое согласование требовалось;
- перед закрытием тикета должны быть приложены все релевантные документы и подтверждения, если без них нельзя понять ход решения.

### Как считать независимые ошибки

- Один и тот же промах не превращай в несколько независимых ошибок только потому, что он затронул цепочку связанных полей.
- Если одна корневая ошибка породила несколько связанных последствий в логировании, по умолчанию считай это одним логическим дефектом.
- Не считай ошибкой сам по себе краткий `AGENT COMMENT`, если ключевые outcome, финансовые действия, refund/payment steps и согласования уже ясно видны в других структурированных зонах тикета и по кейсу остается понятный ход решения.

### Специальный алгоритм для `TYPE / SUBTYPE / SOURCE / CAUSE`

1. Сначала оцени `CAUSE`.
2. Если `CAUSE` обоснованно подходит к ситуации, по классификационной части сразу считай логирование корректным и останавливайся.
3. Если `CAUSE` явно неверен, проверяй `SOURCE`.
4. Только если `CAUSE` уже неверен и `SOURCE` тоже вызывает вопросы, переходи к `TYPE + SUBTYPE` вместе.

Правила этой развилки:

- если `CAUSE` верен, нельзя снижать оценку из-за `TYPE`, `SUBTYPE` или `SOURCE`;
- логирование может опираться как на входящую проблему клиента, так и на итогово выявленную проблему;
- `CAUSE` — это классификационная метка, а не подробное описание механизма;
- если для кейса есть конкретное подходящее значение `CAUSE`, использование `CAUSE = Other` считай ошибкой логирования, а не равноправной альтернативой;
- в `commentary` нельзя предлагать “правильный вариант логирования”, можно только кратко назвать само несоответствие.

### Дополнительные правила по отдельным полям

- `CAUSE = Other`:
  - не используй как безопасный вариант по умолчанию;
  - если по фактам тикета есть более конкретная корректная причина, `Other` нужно считать неправильным логированием;
  - не ослабляй это правило формулировками вроде "лучше выбрать более точный вариант" или "`Other` встречается редко".

- `TIMEFRAME`:
  - оценивай поле с учетом времени в CRM, таймзоны объекта и явных фактов о планируемом времени прибытия;
  - если по тикету видно локальное время объекта и ожидаемое прибытие, не подменяй это только timestamp-ами CRM без пересчета контекста.

- `TPP resolved`:
  - поле должно отражать реальную роль `TPP` в уже решенном кейсе;
  - не ставь его формально только потому, что `TPP` присутствовал в цепочке, если фактически кейс был решен другой стороной.

### Как выбирать score

- `0`:
  - три и более независимые ошибки логирования;
  - или критически неполное оформление, из-за которого по закрытому тикету нельзя надежно восстановить ход решения.
- `1`:
  - две независимые ошибки логирования.
- `2`:
  - логирование и оформление в целом корректны;
  - или есть только одна независимая ошибка оформления, которая не делает кейс непонятным и не тянет на снижение.

### Комментарии для оценки

- Если в `701` есть одна независимая ошибка логирования или оформления, которая сама по себе не снижает итоговую оценку ниже `2`, не оставляй commentary полностью позитивным.
- Не используй это score-`2` исключение для случаев, где комментарий просто можно было бы сделать подробнее, но по структурированным полям, истории и платежным блокам кейс уже читается достаточно ясно.
- В этом случае используй задокументированное score-`2` исключение из `_CORE_instruction.md`: добавь один item в `commentary_plan.score2_exceptions` по `701`.
- Это исключение не меняет саму оценку и не создает `justification`: `701` остается `2`, а `justification` остается `null`.
- В тексте этого exception по `701` не предлагай “правильный вариант логирования”; кратко назови само несоответствие и factual anchor из тикета.

### Пограничные случаи

- Если классификационная часть корректна за счет `CAUSE`, не пытайся “добрать ошибки” через `TYPE` и `SUBTYPE`.
- Если допущена только одна независимая ошибка оформления, итоговая оценка остается `2`.
